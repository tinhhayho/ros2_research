# RMW / middleware swap (Demo 3)

> **Auto-generated by `scripts/demo3_run.sh` from a REAL run.** The *same* node binaries
> (`demo_basic talker` + a counting subscriber) run over three middlewares by flipping one
> env var, `RMW_IMPLEMENTATION`. The RMW is a swappable transport/HAL behind the stable
> `rcl`/`rmw` API — like retargeting firmware from UART to SPI to CAN by swapping the HAL.

## Same-RMW data flow (4 s window, depth 10, domain 42)

| RMW_IMPLEMENTATION | Msgs received | Notes |
|---|---|---|
| rmw_fastrtps_cpp (Fast DDS, default) | 8 | Tier 1 |
| rmw_cyclonedds_cpp (Cyclone DDS) | 8 | Tier 1 |
| rmw_zenoh_cpp (Zenoh) | 8 | available add-on on Jazzy (Tier 1 only from Kilted); needs the `rmw_zenohd` router |

Active-RMW report (`ros2 doctor`) per leg is captured in `docs/img/demo3_active_rmw.txt`.
The `ros2 daemon` caches the RMW from first launch, so each swap runs `ros2 daemon stop` first.

## Cross-RMW interop

| Publisher RMW | Subscriber RMW | Msgs received | Result |
|---|---|---|---|
| Fast DDS | Cyclone DDS | 8 | ✅ **interoperate** — both speak the OMG DDSI-RTPS wire protocol |
| Fast DDS | Zenoh | 0 | ❌ **no data** — Zenoh uses its own wire protocol, not DDSI-RTPS |

Caveats stated honestly: DDS↔DDS interop holds for **topics** with matching QoS/types;
cross-vendor **services/actions are not guaranteed**, and ROS 2 still recommends one RMW
system-wide. DDS↔Zenoh needs a **bridge** (`zenoh-bridge-ros2dds`); they do not discover
each other natively.

## ROS_DOMAIN_ID isolation (same RMW)

| Talker domain | Listener domain | Msgs received | Result |
|---|---|---|---|
| 42 | 99 | 0 | ❌ isolated — different domains never discover each other |

`ROS_DOMAIN_ID` is the canonical lightweight isolation knob and works identically under
host networking — it is how we keep concurrent demos from cross-talking, not container walls.
