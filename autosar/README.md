# AUTOSAR research — Classic vs Adaptive

Self-contained deep-research module of this repo: AUTOSAR **Classic Platform (CP)** and
**Adaptive Platform (AP)** studied separately, then compared — written in English for
embedded/firmware engineers who know MCUs, CAN, and RTOS but are new to automotive
software platforms.

## Layout

```
autosar/
  research/            deliverable research documents (English, URL-sourced)
    notes/             raw per-topic notes from the research agents (provenance)
  blog/                blog-post versions of the research (text-only Markdown;
                       figures are dropped in by hand later)
  specs/               local unmodified mirror of the R25-11 spec PDFs the deck
                       references (offline search; SHA256SUMS + provenance README)
  slides/
    autosar-deck.md    Marp deck TEMPLATE — edit this one
    autosar.md         build artifact (SVGs inlined as base64) — never hand-edit
    autosar.html       build output
    assets/            SVGs specific to this deck; diagrams shared with the main
                       decks resolve from ../../slides/assets/ as a fallback
      drawio-pages/    editable draw.io mirrors — one .xml page per deck-local SVG,
                       same bare-mxGraphModel convention as slides/assets/drawio-pages/
```

## Build

```
./run.sh slides-autosar    # builds autosar/slides/autosar.html (pinned Docker marp-cli)
```

## Rules (inherited from CLAUDE.md)

- Every external claim is URL-sourced; what could not be verified is marked, not shipped.
- New slide content goes through the `fact-check` workflow before it ships.
- `autosar.md` is generated by `scripts/build_slides.py` — never hand-edit it.
